# Result Processing Rules

This chapter introduces how to use rules for post-processing query results, and auxiliary rules used in complex scenarios such as stored procedures.

| Rule | Description |
|:---|:---|
| [`@{resultSet}`](#result-set) | Configure result set mapping strategy (for stored procedures/multiple result sets). |
| [`@{defaultResult}`](#default) | Configure default result set mapping strategy. |

---

## Auxiliary Rules

### RESULT Rule {#result-set}

The `@{resultSet}` rule itself does not affect SQL generation, but in **Stored Procedure** or **Multiple Result Set Query** scenarios, it guides dbVisitor on how to map the returned data.

**Syntax**:
`@{resultSet, key1 = value1, key2 = value2}`

**Supported Scenarios**:
- [Stored Procedure Call](../core/jdbc/procedure)
- [Multi-Result Set Query](../core/jdbc/multi)

**Property Description**:

| Name | Type | Description |
| :--- | :--- | :--- |
| `name` | String | Name of the OUT parameter in the result set. Used to distinguish `INOUT` parameters. |
| `javaType` | Class | Map the result set to a Java type. See [Object Mapping](../core/mapping/about). |
| `rowMapper` | RowMapper | Use custom [RowMapper](../result/for_mapper) to handle mapping. |
| `rowHandler` | RowCallbackHandler | Use custom [RowCallbackHandler](../result/row_callback) to handle each row. |
| `extractor` | ResultSetExtractor | Use custom [ResultSetExtractor](../result/for_extractor) to handle the entire result set. |

---

### DEFAULTRESULT Rule {#default}

`@{defaultResult}` is equivalent to the global default configuration of `@{resultSet}`. When a query returns multiple result sets, you can use this rule to avoid repeating configurations for each result set.

**Syntax**:
`@{defaultResult, key1 = value1, key2 = value2}`

**Property Description**:
Supports `javaType`, `rowMapper`, `rowHandler`, `extractor`, meaning same as above.
